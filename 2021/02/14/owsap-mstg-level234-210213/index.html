LEVEL 2

앞 포스팅에 이어서 LEVEL 2단계 어플리케이션 분석을 진행하겠습니다.

APK 디컴파일 이후 MainActivity 구조를 보면 LEVEL1 단계에서 분석을 했던 것과 크게 차이는 없지만,

두 가지 변경 사항이 있습니다.

  1. Static Library 호출(foo)

  2. AsyncTask 클래스 추가

먼저 1번의 내용부터 보면 코드 내에서 foo라는 라이브러리를 호출하고 있습니다. 이는 라이브러리에서 함수를 정의해서 사용하는 의미가 있습니다.

그리고 AsyncTask의 역할은 백그라운드 작업인데, 디버깅 여부 탐지 로직에 들어간것으로 보아 백그라운드에서 돌아가면서 주기적으로 체크를 하는 것입니다.

아마도 이번 문제는 호출하는 라이브러리까지 타고 들어가서 코드를 분석해야만 풀리지 않을까라는 개인적인 생각을 합니다.

일단 이번앱에서도 Secret String 값을 찾아야 하는데, CodeCheck 메서드를 봤을 때 사진과 같이 smail 코드에선 아무런 내용이 작성되어 있지 않습니다.

다만, native method로 bar() 함수를 호출시켜서 동작하고 있는 것은 알 수 있습니다.

public class CodeCheck {
    private native boolean bar(byte[] bArr);

    public boolean a(String str) {
        return bar(str.getBytes());
    }
}

native method는 간단히, 리눅스 운영체제의 library(so) 파일을 로드해서 사용하는 것인데,

자세한 사항은 아래 링크에 정리가 잘 되어 있어서 참고하면 좋을 것 같다.

  • https://minhyeokism.tistory.com/27

그럼 이 라이브러리가 어디있을까?

APK 압축풀면 lib 폴더에 각 운영체제 환경별로 .so 라이브러리 파일이 있는데, 본인의 환경에 맞는 so만 분석하면 된다.

IDA로 해당 라이브러리 파일을 열어서 CodeCheck 메서드를 확인해보면, 사진과 같이 키 값을 확인할 수 있다.

그런데 후킹을 하지 않고 이렇게 코드 분석으로만 풀 수 있었던가.. 라는 생각을해서 후킹을 통해서도 값을 얻어보기로 했다.

이번에 후킹할 때는 native code로 작성된 것을 후킹해야 하기 때문에 frida의 interceptor 기능을 사용해야 한다.

  • https://frida.re/docs/javascript-api/#interceptor

Interceptor.attach(target, callbacks) 과 같이 사용하는데 target에 so 라이브러리와

callbacks엔 Module.getExportByName()로 메서드를 넣어주면 된다.

        Interceptor.attach(Module.getExportByName("libfoo.so", "strncmp"), {
            onEnter: function(args){
                if(args[2].toInt32() == 23 && Memory.readCString(args[0], 23) == "12345678901234567890123"){
                    console.log("Find Key! : " + Memory.readCString(args[1]));
                }
            }
        })

코드는 그렇게 어렵진 않고, libfoo.so 라이브러리 내 strncmp에 접근해서 onEnter 이벤트가 발생하면, 코드를 실행하도록 구조를 만들었다.

args[2].toInt32() == 23 처리를 해준 이유는 strncmp 함수를 실행하는 조건에 입력값 길이가 23을 만족해야 된다.

그래서 args[0] = 사용자 입력값에 23길이가 채워지면 Memory.readCString() 함수로 메모리에서 값을 읽도록 하였고, 아래와 같이

정상적으로 키 값을 얻어낼 수 있었다.

근데, 중간중간에 디버깅 우회 로직을 안짜놔서 디버깅 탐지도 자꾸 되길래 임시 방편으로다가 후킹을 걸어놓았는데, 단순 임시방편이라..

역시나 Native Method를 사용하여 아까 확인했던 “libfoo.so” 에서 처리하는 구조입니다.

지금부터는 IDA로 넘어가서 libfoo.so에 대한 분석을 진행해야 합니다.

import 항목을 보면 fopen, fork, pthread_create, ptrace 등의 함수가 보입니다.

fork가 있는것으로 보아 자식 프로세스를 만들어서 돌리는 역할을 하는 것 같은데 자세히 분석해봐야 할 것 같구요.

ptrace 함수는 프로세스를 추적하거나 실행되는 기능을 관리하는 함수로 디버깅 체크에 사용될 듯 합니다.

IDA로 sub_30D0 함수를 추적하여 디 컴파일링 해보면 이렇게 원본 코드가 출력됩니다.

fopen() 함수로 /proc/self/maps를 읽어서 frida, xposed 문자열을 찾고 있습니다.

while ( !strstr(v1, "frida") && !strstr(v1, "xposed") );

해당 조건에서 frida가 탐지될 경우 Tampering detected! Terminating... 라는 메세지를 출력하고 있네요.

마지막 부분은 예외처리를 하는 부분인데 /proc/self/maps 이걸 못 읽으면 goodbye() 함수를 실행시켜 종료시켜버립니다.

void __noreturn goodbye(void)
{
  raise(6);
  _exit(0);
}

우선 탐지하는 로직을 우회하기 위해서는 기존에서 풀이하던 방식으로는 불가능하다.

왜냐하면, 이미 실행된 앱(프로세스)에서는 탐지 로직이 돌고 있기 때문에, 후킹을 건다해도 앱을 종료시켜버린다.

따라서 앱이 실행되는 시점이 아닌, 앱이 로드되는 시점에 후킹을 걸어서 탐지 로직보다 먼저 코드를 인젝션 시켜야한다.

그에 대한 방법으로 아래 사항을 정리한다.

https://frida.re/news/2018/05/05/frida-11-0-released/

frida에서는 device.spawn 이라는 기능을 제공하며, 해당 기능은 프로세스를 재 실행해주는 기능이다.

이 spawn 기능을 사용하여 APP의 pid를 얻어오고 attach 과정을 거친 뒤, resume 기능으로 프로세스를 실행시켜주면

앱이 로드되는 시점에 후킹이 가능하다.

spawn과 resume에 대해서는 아래 링크를 참고해도 좋을 것 같다.

https://bachs.tistory.com/entry/iOS-Hooking2Frida

if __name__ == "__main__":
    try:
        process = frida.get_usb_device(timeout=5)
        pid = process.spawn(APP_NAME)
        run = process.attach(pid)
        process.resume(pid)
        script = run.create_script(hook)
        script.on('message', on_message)
        print("[*] Start Injection")
        script.load()
        sys.stdin.read()
        process.detach()
    except Exception as e:

이제, 위에서 분석했던 내용대로 라이브러리에 접근하여 strstr 함수를 후킹한 뒤,

반환하는 값을 0으로 만들어주면 탐지 로직이 돌더라도 우회가 되는 것을 확인할 수 있을 것이다.

우선, 라이브러리 파일 내에 있는 함수를 후킹하기 위해서는 frida에서 Interceptor.attach 기능을 제공하고 있다.

해당 기능에 대한 설명은 이전에 블로그에 정리한 글과 frida 공식 문서인 아래 링크를 참고하기 바란다.

https://blog.jhyeon.dev/2021/02/22/frida-hook1-210222/

https://frida.re/docs/javascript-api/#interceptor

frida에서 Interceptor.attach를 사용하면 특정 라이브러리 내 함수에 접근할 수 있고 후킹을 통해 내가 원하는 방식으로

앱이 동작하도록 할 수 있다.

    Interceptor.attach(Module.getExportByName("libc.so", "strstr"), {
        onEnter: function(args){
            var haystack = Memory.readUtf8String(args[0]);
            if(haystack.indexOf("frida") !== -1 || haystack.indexOf("xposed") !== -1){
                this.frida = Boolean(1);
            }
        },

        onLeave : function(retval){
            if(this.frida){
                retval.replace(0);
            }
            return retval;
        }
    });

    Java.perform(function(){
        var sys = Java.use("java.lang.System");
        sys.exit.overload("int").implementation = function(var_0){
            send("System.exit() Hooked!");
        }
    });

IDA에서 탐지 로직을 본 것 처럼 frida, xposed가 탐지되는 경우 0을 반환하도록만 처리해줬다.

또한 exit가 호출될 때 강제 종료가 안되기 위해서 system->exit() 를 후킹하였다.

위와 같은 코드를 작성하고 테스트해보면 아래와 같이 후킹이 정상적으로 걸려서 우회한 것을 볼 수 있다.
